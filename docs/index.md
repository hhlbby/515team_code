# Глоссарий проекта «Туристическое бюро»

---

## База данных (Database)
Центральное хранилище информации о пользователях, автобусах, экипажах, маршрутах и рейсах. В проекте реализована как класс `Database`, который содержит векторы объектов.

## Пользователь (User)
Объект, представляющий сотрудника системы. Содержит:
- имя пользователя (`username`),  
- пароль (`password`),  
- роль (`role`) — администратор (`admin`) или член экипажа (`crew`).  

Отвечает за **аутентификацию и авторизацию** в системе.

## Аутентификация (Authentication)
Процесс проверки логина и пароля пользователя для входа в систему.

## Авторизация (Authorization)
Процесс определения уровня доступа пользователя к функционалу:
- администратор — полный доступ;  
- член экипажа — доступ только к своим данным.

## Автобус (Bus)
Средство передвижения для туристов. Основные атрибуты:
- номер автобуса,  
- название модели,  
- суммарный пробег (`mileage`).  

Каждому автобусу **закреплён один экипаж**.

## Экипаж (Crew)
Группа сотрудников, обслуживающих один автобус. Состоит из **трёх членов** (`CrewMember`).  
- Имеет ссылку на автобус, которым управляет.  
- Начисления за рейсы распределяются между членами экипажа.

## Член экипажа (CrewMember)
Сотрудник, входящий в состав экипажа. Атрибуты:
- ФИО,  
- табельный номер,  
- опыт работы (`experience`),  
- категория,  
- адрес,  
- год рождения.  

Может просматривать только свои данные и начисления.

## Маршрут (Route)
Экскурсионный маршрут с информацией:
- название маршрута,  
- начальный и конечный пункт,  
- протяжённость пути (`distance`).  

Используется для планирования рейсов.

## Рейс (Trip)
Выполненное путешествие по маршруту. Атрибуты:
- автобус,  
- маршрут,  
- дата отправления и прибытия,  
- количество пассажиров,  
- стоимость билета.  

Рейс формирует **начисление дохода экипажу**.

# Дополнительная спецификация проекта «Туристическое бюро»

---

## 1. Производительность
- Система должна обрабатывать до 1000 рейсов и 500 автобусов без заметного снижения производительности.  
- Время отклика на запрос аутентификации пользователя не должно превышать 1 секунды.  
- Отчёты по рейсам и начислениям должны формироваться за <2 секунды при объёме данных до 500 записей.

## 2. Надёжность и отказоустойчивость
- Система должна корректно обрабатывать ошибки ввода данных (например, отрицательное количество пассажиров или пустые поля).  
- Все исключения в коде должны быть обработаны для предотвращения аварийного завершения программы.  
- При ошибке доступа к данным пользователь получает информативное сообщение.

## 3. Безопасность
- Пароли пользователей хранятся в памяти в зашифрованном виде (в будущей версии — хэширование).  
- Только аутентифицированные пользователи могут получить доступ к функционалу системы.  
- Члены экипажа могут просматривать только свои рейсы и начисления; администратор — полный доступ.  

## 4. Интероперабельность
- Приложение консольное, работает в среде Windows и Linux.  
- Возможна интеграция с будущей базой данных (например, SQLite или MySQL) через изменение класса `Database`.

## 5. Пользовательский интерфейс
- Интерфейс консольный.  
- Понятные подсказки для ввода команд и выбора действий.  
- Для администраторов выводятся списки автобусов, экипажей, рейсов и отчетов.  
- Для членов экипажа — доступ только к своим рейсам и начислениям.

## 6. Локализация
- Текущая версия поддерживает только русский язык.  
- Все сообщения в консоли и названия команд на русском языке.  

## 7. Аппаратные требования
- Процессор: 1 ГГц или выше.  
- ОЗУ: 2 ГБ.  
- Место на диске: 50 МБ (для проекта и временных файлов).  

## 8. Ограничения
- Данные хранятся только в оперативной памяти; после закрытия программы все данные теряются (в будущем планируется сохранение в файлы/БД).  
- Одновременная работа нескольких пользователей не поддерживается.  
- Нет графического интерфейса; все взаимодействия через консоль.  

## 9. Журналирование
- В текущей версии система выводит сообщения об ошибках и успешных операциях в консоль.  
- В будущем возможно ведение лог-файлов для аудита действий пользователя.  

## 10. Расширяемость
- Добавление новых сущностей (например, дополнительных типов маршрутов или автобусов) возможно без изменения существующих классов.  
- Можно расширить функционал начислений, процентных ставок и отчетов через новые методы классов.  
- Возможна интеграция с графическим интерфейсом или веб-приложением.

## 1. USERS (Пользователи)
Таблица для хранения учетных записей пользователей приложения.

| Поле           | Тип       | Описание                                      |
|----------------|-----------|-----------------------------------------------|
| user_id        | INTEGER   | Уникальный идентификатор, PRIMARY KEY         |
| username       | TEXT      | Логин пользователя, UNIQUE, NOT NULL          |
| password_hash  | TEXT      | Хэш пароля, NOT NULL                           |
| full_name      | TEXT      | ФИО пользователя, NOT NULL                     |
| role           | TEXT      | Роль: driver, guide, admin, manager, CHECK   |
| created_at     | TIMESTAMP | Дата создания записи, DEFAULT CURRENT_TIMESTAMP |

---

## 2. BUSES (Автобусы)
Информация об автобусах, используемых для рейсов.

| Поле           | Тип       | Описание                                      |
|----------------|-----------|-----------------------------------------------|
| bus_id         | INTEGER   | Уникальный идентификатор, PRIMARY KEY        |
| bus_number     | TEXT      | Номер автобуса, UNIQUE, NOT NULL              |
| model_name     | TEXT      | Модель автобуса, NOT NULL                     |
| total_mileage  | INTEGER   | Общий пробег автобуса в км, DEFAULT 0         |

---

## 3. CREW_MEMBERS (Члены экипажа)
Информация о членах экипажа, которые строго закреплены за автобусом.

| Поле           | Тип       | Описание                                      |
|----------------|-----------|-----------------------------------------------|
| crew_id        | INTEGER   | Уникальный идентификатор, PRIMARY KEY        |
| tax_number     | TEXT      | Табельный номер, UNIQUE, NOT NULL             |
| last_name      | TEXT      | Фамилия, NOT NULL                             |
| first_name     | TEXT      | Имя, NOT NULL                                 |
| patronymic     | TEXT      | Отчество                                      |
| birth_year     | INTEGER   | Год рождения, NOT NULL                        |
| address        | TEXT      | Адрес                                         |
| experience_years | INTEGER | Стаж в годах, DEFAULT 0                       |
| category       | TEXT      | Категория (водитель, гид, механик и т.п.), NOT NULL |
| bus_id         | INTEGER   | Ссылка на автобус, FOREIGN KEY BUSES(bus_id)  |
| user_id        | INTEGER   | Связь с учетной записью, UNIQUE, FOREIGN KEY USERS(user_id) |

**Индексы:**  
- `idx_crew_bus` — ускоряет выборку по автобусу

---

## 4. ROUTES (Маршруты)
Информация о маршрутах.

| Поле           | Тип       | Описание                                      |
|----------------|-----------|-----------------------------------------------|
| route_id       | INTEGER   | Уникальный идентификатор, PRIMARY KEY        |
| route_name     | TEXT      | Название маршрута, NOT NULL                   |
| start_point    | TEXT      | Начальный пункт маршрута, NOT NULL            |
| end_point      | TEXT      | Конечный пункт маршрута, NOT NULL             |
| distance_km    | REAL      | Длина маршрута (км), > 0, NOT NULL           |

---

## 5. TRIPS (Выполненные рейсы)
Информация о выполненных рейсах.

| Поле           | Тип       | Описание                                      |
|----------------|-----------|-----------------------------------------------|
| trip_id        | INTEGER   | Уникальный идентификатор, PRIMARY KEY        |
| bus_id         | INTEGER   | Ссылка на автобус, FOREIGN KEY BUSES(bus_id) |
| route_id       | INTEGER   | Ссылка на маршрут, FOREIGN KEY ROUTES(route_id) |
| departure_date | DATE      | Дата отправления, NOT NULL                    |
| arrival_date   | DATE      | Дата прибытия, >= departure_date             |
| passengers_count | INTEGER | Количество пассажиров, > 0                   |
| ticket_price   | REAL      | Стоимость билета, >= 0                        |
| total_cost     | REAL      | Автоматически рассчитываемая стоимость = passengers_count * ticket_price |

**Индексы:**  
- `idx_trips_bus` — ускоряет выборку рейсов по автобусу  
- `idx_trips_route` — ускоряет выборку по маршруту  
- `idx_trips_departure` — ускоряет выборку по дате отправления

---

## 6. CONFIG (Конфигурация приложения)
Хранит гибкие настройки системы.

| Поле           | Тип       | Описание                                      |
|----------------|-----------|-----------------------------------------------|
| config_key     | TEXT      | Ключ настройки, PRIMARY KEY                  |
| config_value   | TEXT      | Значение настройки, NOT NULL                  |
| description    | TEXT      | Описание настройки                             |
| updated_at     | TIMESTAMP | Дата последнего обновления, DEFAULT CURRENT_TIMESTAMP |

---

## Связи между таблицами (ER)
USERS(user_id) <-- CREW_MEMBERS(user_id)
BUSES(bus_id) <-- CREW_MEMBERS(bus_id)
BUSES(bus_id) <-- TRIPS(bus_id)
ROUTES(route_id) <-- TRIPS(route_id)


**Описание связей:**
- `BUSES 1:N CREW_MEMBERS` — один автобус может иметь несколько членов экипажа  
- `USERS 1:1 CREW_MEMBERS` — член экипажа может быть привязан к учетной записи  
- `BUSES 1:N TRIPS` — один автобус может выполнять несколько рейсов  
- `ROUTES 1:N TRIPS` — один маршрут может быть использован в нескольких рейсах  

---

## Индексы
- `idx_crew_bus` — ускоряет выборку членов экипажа по автобусу  
- `idx_trips_bus` — ускоряет выборку рейсов по автобусу  
- `idx_trips_route` — ускоряет выборку рейсов по маршруту  
- `idx_trips_departure` — ускоряет выборку рейсов по дате отправления

# Функциональные требования проекта «Туристическое бюро»

---

## 1. Аутентификация и авторизация
- Система должна обеспечивать вход пользователей по логину и паролю.  
- Поддержка двух ролей:
  - **Администратор** — полный доступ ко всем функциям системы.  
  - **Член экипажа** — ограниченный доступ, просмотр только своих данных и начислений.  

## 2. Управление пользователями
- Администратор может создавать, редактировать и удалять пользователей системы.  
- Пользователи хранят информацию о логине, пароле и роли.  

## 3. Управление автобусами
- Добавление, редактирование и удаление автобусов.  
- Каждый автобус имеет уникальный номер, название и суммарный пробег.  
- Автобус связан с одним экипажем.  

## 4. Управление экипажами
- Администратор может создавать, редактировать и удалять экипажи.  
- Экипаж состоит из трёх членов (`CrewMember`).  
- Каждый экипаж привязан к определённому автобусу.  

## 5. Управление членами экипажа
- Добавление, редактирование и удаление членов экипажа.  
- Для каждого члена хранится: ФИО, табельный номер, стаж, категория, адрес, год рождения.  
- Члены экипажа могут просматривать только свои данные.  

## 6. Управление маршрутами
- Добавление, редактирование и удаление маршрутов.  
- Маршрут содержит: название, начальный и конечный пункт, протяжённость пути.  

## 7. Управление рейсами
- Добавление, редактирование и удаление рейсов.  
- Рейс содержит: автобус, маршрут, дату отправления и прибытия, количество пассажиров, стоимость билета.  
- После добавления рейса автоматически рассчитываются **начисления экипажа** (процент от стоимости рейса).  

## 8. Расчет и учет начислений
- Система автоматически рассчитывает доход экипажа за выполненные рейсы.  
- Член экипажа может просматривать **только свои начисления**.  
- Администратор может просматривать доходы всех экипажей за выбранный период.  

## 9. Отчеты и статистика
- По указанному автобусу:
  - Перечень выполненных рейсов за выбранный период.  
  - Общее количество поездок, перевезённых пассажиров и полученные деньги.  
- По каждому экипажу:
  - Количество начисленных денег за выбранный период.  
- По наиболее дорогому маршруту:
  - Сведения об автобусах, экипажах и стоимости билетов.  
- По автобусу с наибольшим суммарным пробегом:
  - Количество перевезённых пассажиров и величина пробега.  

## 10. Ограничения доступа
- Администратор — полный доступ ко всем операциям и отчетам.  
- Член экипажа — доступ только к своим данным и своим начислениям.  

## 11. Валидация данных
- Система проверяет корректность вводимых данных при добавлении:
  - Рейсов — соответствие автобусу и маршруту.  
  - Начислений — процент от стоимости рейса.  
  - Даты и числовых значений (например, количество пассажиров, пробег).  

## 12. Архитектурные требования
- Консольное приложение на языке C++.  
- Проект многомодульный, с папками: `src`, `includes`, `bin`, `build`.  
- Используются отдельные классы для автобусов, маршрутов, экипажей, рейсов и пользователей.  
- Все операции выполняются через объекты и методы классов, без прямого взаимодействия с файловой системой (в текущей версии — данные в памяти).
